# Writing an OpenTakeoff feature plugin (v1 contract)

A feature plugin is a folder under `web/src/features/<name>/` exporting a
**default descriptor** from `plugin.js` (or `.jsx`). Public core ships none; a
fork drops feature folders in, and every upstream merge leaves them alone. The
contract is the pure TypeScript core under `web/src/lib/plugins/` ‚Äî its published
types live in `web/public-types/` (generated by `npm run emit-types`). Import the
descriptor and context types from there.

> **Slice status.** Slice 2a ships the *contract core* only: the descriptor
> schema, `validateDescriptor`, the `buildCanvasContext` fa√ßade, device storage,
> the version gate, and the registry/loader. The React host that renders overlays
> and export slots lands in slice 2b ‚Äî until then a descriptor is loaded and
> validated but not yet mounted.

## The descriptor (frozen, exactly four keys)

```js
export default {
  id: "my-feature",          // unique, non-empty; `__*__` ids are reserved
  minCtxVersion: "1.0",      // "major.minor" ‚Äî the oldest context you run on
  overlays: [ /* OverlaySlot[] */ ],
  exports:  [ /* ExportSlot[]  */ ],
};
```

`validateDescriptor` **rejects any key outside `{ id, minCtxVersion, overlays,
exports }`.** `panels` / `setup` / `agentTools` are explicitly out of v1; they
arrive by a *minor* context bump when a real consumer lands. A descriptor pinned
to a *future* `minCtxVersion` is **version-skipped** ("host too old"), never
hard-rejected ‚Äî so you can ship a plugin that targets a newer host and older
hosts simply skip it.

> **Ordering matters if you call the primitives yourself.** That forward-compat
> guarantee lives in the *order* the host runs things: `selectRenderablePlugins`
> (the version gate) FIRST, then `validateDescriptor` on only the surviving set.
> `validateDescriptor` is structural and host-version-blind ‚Äî call it directly on
> a future-pinned descriptor and it will hard-reject the unknown key it carries.
> Use `loadFeaturePlugins`, which composes the two in the correct order; reach for
> the primitives only if you're re-implementing the loader.

### Overlay slots ‚Äî render-time `render({ ctx, onClose })`

```js
overlays: [{
  id: "summary",
  label: "Summary",
  icon: "‚àë",                                   // optional
  render: ({ ctx, onClose }) => <MyPanel ctx={ctx} onClose={onClose} />,
}]
```

> **The host owns placement ‚Äî render RELATIVE content, don't self-position.** Your
> `render()` output is dropped into a host-positioned, width-bounded slot (a
> ~300px panel that expands upward, clear of the canvas's own controls). Do **not**
> set `position: fixed`/`absolute` with your own `top`/`left`/`bottom` on the root
> ‚Äî that escapes the safe zone and can cover app controls. Style the panel's
> look (background, border, padding) but let the host place it. At most one
> overlay is open at a time, and it shares its slot with the plugin manager.

### Export slots ‚Äî action-time **void** `onSelect(ctx)`

```js
exports: [{
  id: "summary-txt",
  label: "Summary (TXT)",
  onSelect: (ctx) => {
    const text = buildSummary(ctx.getConditions(), ctx.getShapes(), ctx.units);
    ctx.download("summary.txt", text, "text/plain"); // the slot does its own download
  },
}]
```

`onSelect` returns **nothing** ‚Äî it performs its own `ctx.download`. (This
diverges from the throwaway spike's returning `run(ctx)`; use `onSelect`.)

## The canvas context (`ctx`) ‚Äî frozen surface

A plugin never touches canvas internals, React state, or the store directly. It
gets a `ctx` with **exactly** these members:

| member | shape |
| --- | --- |
| `version` | `"major.minor"` string |
| `units` | `"imperial"` / `"metric"` |
| `getConditions()` | live conditions array |
| `getShapes()` | live shapes array |
| `getActiveConditionId()` | id or `null` |
| `getSelectedShapeId()` | id or `null` |
| `getProjectName()` | string |
| `commands.dispatchShape(cmd, opts?)` | routes through the real undo/redo + provenance chokepoint |
| `storage` | per-plugin key/value store (below) |
| `download(filename, text, mime?)` | browser download |

Accessors read **live** values ‚Äî call them when you need the current state, don't
cache the result across renders.

## `ctx.storage` ‚Äî per-plugin key/value

```js
await ctx.storage.set("note", "hello");       // scope defaults to "device"
const v = await ctx.storage.get("note");
await ctx.storage.remove("note");
```

- Keys are namespaced to your plugin; you cannot read another plugin's keys.
- `scope` is an **input** (`{ scope: "device" }`), defaulting to `"device"`.
- `scope: "project"` is **not yet supported** in v1 and rejects the returned
  promise (`await`/`.catch` it). It is device-local only for now.
- The handle exposes no property revealing how it is backed ‚Äî treat it as an
  opaque async key/value store.

## Versioning & deprecation

The context is versioned `major.minor`. A **minor** bump is additive ‚Äî a new
accessor or descriptor slot ‚Äî and an existing plugin keeps working. A **major**
bump is reserved for a breaking change and strands plugins pinned to the prior
major until they opt in by raising `minCtxVersion`.

**Deprecation story:** a member is never silently removed within a major. When a
member is to be retired, its replacement ships first in a minor bump (both live
side by side for at least one minor), the old one is documented as deprecated,
and it is only removed at the next **major** ‚Äî at which point plugins still
pinned to the old major continue to load against the old host and are
version-skipped by new hosts rather than breaking mid-session. Pin the *lowest*
`minCtxVersion` your plugin actually needs so a host upgrade never gates you out.
