Design exploration · Takeoff toolbar · 4 options

One row is doing six different jobs.

The current toolbar interleaves session, file, tools, drawing aids, scale, and outputs in a single wrapping row — so related controls sit far apart, and the wrap point shifts as buttons appear and disappear. Below: an audit of the problem, then four redesigns as clickable mockups. Every dropdown, toggle, and popover works.

The audit — what the row is actually holding

Color-coding each control by its concern makes the problem visible: the sequence changes concern nine times across one row. There is no place your eyes learn to go.

− No spatial logic“Sign out” sits one button away from “Open.” Report and Snapshots — the payoff of the whole workflow — end up wherever the wrap drops them.
− The bar moves under your handsUngroup, Regroup, the fill-sensitivity slider, Finish, and Create all appear conditionally mid-row, pushing everything after them and shifting the wrap point. Muscle memory never settles.
− Scale is a setting dressed as a toolThe scale picker, the “plan says” chip, calibrate, and Hi-Res are per-sheet state, but they occupy prime tool real estate — four controls for something you touch once per sheet.
− Three competing “primaries”Ink-filled Open, ink-filled REPORT, and the cobalt armed tool all claim top visual priority, plus a green Snap and a red scale warning. Nothing wins.

One thing the current bar already gets right, and every option below keeps: the ToolMenu pattern — a dropdown whose face shows the armed tool (STACK-style). The redesigns extend that pattern to more of the bar rather than inventing a new one.

A

Two decks — project bar over work bar

Split by frequency, not by feature: things you do to the project (open, navigate, export, account) live in a slim top deck; things you do to the sheet (arm tools, toggle aids, set scale) live in a taller work deck with drafting-style group captions. Nothing hides — it just gains a fixed home.

B

One consolidated row — dropdowns do the grouping

The most aggressive de-clutter: one 40 px row that never wraps. Aids collapse into a checkable menu whose face counts what’s on; the four scale controls become a single status chip that opens a popover; file verbs and the session move under a File menu. Maximum canvas, minimum chrome.

C

Left tool rail — tools live beside the work

The Bluebeam-shaped answer. Drawing tools and aids move to a 46 px vertical rail with flyout menus; the top bar keeps only project concerns. Vertical space is nearly free on a plan-reading screen, horizontal space is precious — this trades the cheap one for the expensive one, and tool clicks land next to where you draw.

R

Recommended — A’s structure, B’s consolidations

Two decks, but with the three highest-value merges from B: an account chip, a single scale chip with popover, and Hi-Res + fill sensitivity folded into a small render menu. High-frequency toggles (Snap, 45°) stay visible — one click, fixed position. Transient actions (Finish, Create, “click the opposite corner…”) get a reserved slot at the end of the work deck so nothing ever shifts.

Whichever direction wins, three principles from this exercise are worth adopting regardless: one home per concern (a control’s position tells you what it is), state changes may never move other controls (conditional UI gets reserved space or lives in a menu), and menu faces carry state (armed tool, aids count, scale value) so consolidation never hides what’s on.
